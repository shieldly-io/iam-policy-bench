# iam-policy-bench

A neutral, reproducible benchmark of AWS IAM policy security scanners against a small public
corpus of policies with documented ground truth. Maintained by [Shieldly](https://www.shieldly.io),
which is also one of the tools being measured — every result below is generated by running the
actual tools, not asserted.

## Run it yourself

```bash
npm install
pip install -r requirements.txt   # parliament, cloudsplaining
npm run bench
```

`run.js` runs [`@shieldly/iam-lint`](https://www.npmjs.com/package/@shieldly/iam-lint) in-process
(it's a plain JS import) and shells out to
[Parliament](https://github.com/duo-labs/parliament) and
[Cloudsplaining](https://github.com/salesforce/cloudsplaining) for the other two. Nothing is
mocked or pre-recorded.

## Methodology

Each file in `policies/` is a real IAM policy document with a documented expected classification
in `ground-truth.json` — `vulnerable` (with a specific category) or `clean`. The harness runs
every tool against every policy and records only whether the tool raised **any** finding, not how
many.

**Why detection, not counts:** the three tools report findings at very different granularity —
Parliament with community auditors enabled can emit 20+ individual findings for one wildcard
statement (one per privilege-escalation method it separately recognizes), while Cloudsplaining
groups the same statement into a handful of categorized issues. Comparing raw counts would
reward whichever tool happens to split findings more finely, not whichever tool actually caught
the problem. Binary detection (did it flag *anything* on this bad policy? did it stay silent on
this clean one?) is the fair, comparable signal.

## Results (last run: 2026-07-15)

| Policy | Ground truth | @shieldly/iam-lint | Parliament | Cloudsplaining |
|---|---|---|---|---|
| `01-full-admin-wildcard.json` (Unrestricted access (Action:\* / Resource:\*)) | vulnerable | ✓ | ✓ | ✓ |
| `02-passrole-lambda-escalation.json` (Privilege escalation (iam:PassRole + Lambda)) | vulnerable | ✓ | ✓ | ✓ |
| `03-iam-createpolicyversion-escalation.json` (Privilege escalation (iam:CreatePolicyVersion)) | vulnerable | ✓ | ✓ | ✓ |
| `04-s3-data-exfil-wildcard-resource.json` (Data exfiltration risk (s3:GetObject on Resource:\*)) | vulnerable | ✓ | ✓ | ✓ |
| `05-clean-scoped-dynamodb.json` (Scoped to specific actions + a single resource ARN) | clean | ✓ | ✓ | ✓ |
| `06-clean-lambda-logging.json` (Scoped to specific actions + a single resource ARN) | clean | ✓ | ✓ | ✓ |

All three tools currently score 6/6 on this corpus — a genuine tie, not a typo. That's an honest
result for a first, deliberately basic set of cases: all three tools correctly catch obvious
wildcards, known privilege-escalation action combos, and stay quiet on tightly-scoped policies.
The value of this project isn't in today's numbers, it's the reproducible harness + growing
corpus — differences show up as trickier, more realistic policies get added (see Contributing).

## Known scope differences (not run, and why)

- **Checkov, cfn-nag**: both operate on CloudFormation/Terraform *templates*, not standalone IAM
  policy JSON — a policy would need to be wrapped in a minimal CFN resource first. Out of scope
  for this corpus format; a separate CFN-template corpus is a reasonable follow-on.
- **AWS IAM Access Analyzer (`ValidatePolicy`)**: requires a real AWS account/API access, which
  this reproducible-by-anyone harness deliberately avoids depending on.
- Parliament and Cloudsplaining are both designed for **identity** policies. Resource-based-policy
  checks (e.g. a wildcard `Principal` with no `Condition` on an S3 bucket policy) are outside
  their scope by design — `@shieldly/iam-lint` and Shieldly's full analyzer cover that case, but
  it isn't in this corpus yet since it wouldn't be a fair three-way comparison.
